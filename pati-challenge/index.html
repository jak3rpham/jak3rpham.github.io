Case 01 — Creative Orchestration

Brief to ad-ready in six hours, by routing each stage to the right model.

Pipeline shape · Branching → Converging · Tools: Claude · Flux Pro · GPT Image · Seedance
Bottleneck

A creative brief for a fictional mobile RPG game ad. Six hours of self-imposed constraint: blank input, ad-ready output expected. The default path — prompt one model into doing everything — caps out fast. Text-to-image models lose composition fidelity at scale, image-to-video tools need clean keyframes to work with, copy generators don't think visually.

The bottleneck is not "use AI." It is: what tool, at what stage, with what handoff between them.

Architecture
Full Weavy canvas with workflow diagram · model selection rationale · iteration evidence — at jak3rpham.github.io/bong-vespera ↗
Workflow
01

Concept decomposition Claude

Brief → 5 concept directions (visual codes, emotional register, audience cue). Pick 1, lock the visual codes that downstream stages will inherit.

02

Storyboard & copy Claude

Shot list, copy hooks, technical keyframe descriptions written in a format the next stage's models can ingest directly. Each frame description includes composition, lighting, and mood tokens.

03

Hero keyframes Flux Pro

Five high-fidelity keyframes generated from the locked descriptions. Flux Pro picked here because it holds composition and atmosphere best at this scale — expensive per call but the cleanest hero output.

04

Vietnamese typography GPT Image

Title cards and Vietnamese display copy. Picked here because GPT Image handles Vietnamese diacritics cleaner than Flux at this scale — the kind of detail that matters when the audience is local and "almost-right" typography reads as foreign-made ad.

05

Motion samples Seedance

Locked keyframes → motion clips. Seedance is doing the one thing image-only models can't, with a clean keyframe input it actually has something to work with.

06

Assembly Manual

Final cut, copy overlay, sound design. The judgment call about what survives and what gets cut is human — AI is upstream of taste, not a replacement for it.

Why this routing
The decision behind the architecture — and the reroute mid-pipeline

Every model has a sweet spot. Flux Pro's is hero fidelity. GPT Image's is Vietnamese typography. Seedance's is motion. Claude's is the reasoning layer that keeps prompts coherent across handoffs so each downstream model gets clean inputs.

What actually happened mid-build: Flux character consistency broke between keyframes — same character, different frames, the model lost the face. The decision wasn't to abandon the concept. It was to reroute: lock the character codes more tightly in Claude's prompt layer, regenerate, accept the cost. That reroute is the work. A single-model pipeline doesn't give you that escape hatch — you either ship the inconsistency or restart from zero. Routing across specialists means you can fail at one stage without burning the whole run.

Case 02 — Bias-Reduced Research Pipeline

Top experts — measured, not guessed. An 8-phase research pipeline.

Pipeline shape · Sequential phases · Bias reduction by cross-reference · Tools: DataForSEO · YouTube API v3 · Supadata · Claude API · Python in Cursor
Bottleneck

A research challenge: identify the top 10 voices in AI-powered SEO content — a niche covering SEO, AEO, GEO, and LLMO. The naive approach (list the people I follow) fails three ways. It's biased toward who's loud in my feed, not who's actually shaping the field. It's static — no signal on who's recently active versus who's coasting on past reputation. And it can't be defended — when the receiver asks "why these ten," there's no answer except "I picked them."

The bottleneck wasn't writing the list. It was building a process where the list comes from measurable signals, where the result is reproducible, where bias is reduced by cross-referencing independent sources, and where every exclusion has a reason.

Architecture
n8n workflow canvas · 8 phases · DataForSEO → YouTube → scoring → cross-reference → Supadata → Claude API audit
Workflow
P1

Keyword validation DataForSEO API

Test 15 seed keywords against US Google search volume. Keep 4 with real community traction (AI SEO 8,100/mo, GEO SEO 2,400/mo, AI search optimization 1,000/mo, LLM SEO 880/mo). Drop terms that read accurate but lack volume — "AI content production" sounds right and gets 10/mo. Validation prevents searching against vocabulary nobody actually uses.

P2

YouTube channel discovery YouTube Data API v3

Search the four validated keywords. Output: 38 unique channels surfaced. The discovery layer is intentionally wide — over-include here, filter downstream.

P3

Channel scoring Engagement · Recency · Subs

Weighted scoring: engagement rate (35%) + recency over 90 days (30%) + subscriber base (35%). Auto-exclude fraud signals — channels where average views fall under 10 relative to subscriber count. Threshold-based, not vibes-based.

P4

Relevance scoring Keyword match

Keyword matching against video titles and descriptions. Auto-exclude zero engagement and clearly off-topic channels. Removes the noise that pure subscriber-count surveys keep.

P5

Cross-reference for bias reduction LinkedIn · Newsletter · Speaker lists

This is the phase that does the most work. Cross-reference against ≥2 independent industry sources: BrightonSEO, MozCon, SEO Week speaker lists, plus practitioner newsletter rankings. YouTube alone surfaces who YouTube rewards (frequent video format, often US-centric); cross-referencing pulls in voices that publish primarily in long-form or talk circuits. Merged scoring across platforms reduces single-platform bias.

P6

Content collection Supadata API

YouTube transcripts pulled programmatically via Supadata API. Blog and newsletter articles via browser collection. Output is structured: one JSON per expert containing transcript text, video metadata, article URLs, dates. Consistent shape across sources is what makes downstream auditing feasible.

P7

Targeted expansion Interview discovery

Some experts are quiet on their own channels but loud on others' shows. Interview-appearance discovery catches them. Without this phase, the pipeline misses voices like Koray Tuğberk — founded the Topical Authority methodology, primarily long-form on holisticseo.digital, quiet on YouTube directly but heavily cited in industry interviews.

P8

Quality audit gate Claude API

Claude API audits the collected content for relevance to the niche, recency, and depth. Final filter before the list locks. The gate is where AI judgment lives in this pipeline — everything upstream is deterministic scoring; this phase is where qualitative reasoning happens, but only after the data has been narrowed to a reviewable set.

Why this design
The architecture is built around two failure modes

Bias from a single platform. YouTube alone surfaces who YouTube rewards — frequent video output, often US-centric, often optimized for the algorithm rather than depth. P5's cross-reference against LinkedIn and newsletter speaker lists pulls in voices that publish primarily in other formats. Merged scoring is what produces a list that holds up across communities, not just one.

Bias from prior knowledge. If I'd just listed "experts I think are top," the list would have been who I follow. The pipeline is designed to find people I wouldn't have found, AND to filter out people who'd survive my prior bias but fail the data — channels with high subscriber counts and zero recent depth, voices loud in my feed but absent from speaker circuits.

The output isn't "Tatsuki's opinion of top experts." It's a list defensible to a stranger who asks "why these ten." Every exclusion has a phase number attached.

Case 03 — Analytics-to-Decision

AI designed the contracts. n8n executes them. Determinism by design.

Pipeline shape · AI in design-time · Locked deterministic runtime · Tools: Custom MCP servers · n8n · GitHub · Static template
Bottleneck

Stage A — the obvious bottleneck. Marketing data fragmented across GSC, GA4, CrUX, PageSpeed. Each platform has its own dashboard, its own latency, its own anomaly definitions. Manual aggregation = 4–6 hours per week of mechanical work where human judgment adds zero value, and by the time the report is ready, the data is already stale.

Stage B — the bottleneck I only found after building v1. With Claude in the rendering loop, every weekly report came out subtly different: section order shifted, threshold descriptions used different phrasing, layout breathed differently each run. For internal stakeholders who scan reports weekly, this is anti-usability — the cognitive cost becomes "where do I look this week" instead of "what changed this week." The bug wasn't in the analysis. It was in letting an LLM render a recurring artifact that needs to look the same.

The diagnosis: LLM non-determinism is a feature for analysis (different angles each week) and a bug for layout (must be the same each week). The architecture had to separate the two.

Architecture
n8n workflow canvas · Schedule → 4 parallel MCP fetches → Claude API (structured output, slot-fill only) → JSON compose → dual GitHub commit
Workflow
DT

Design-time · Build with Claude One-time

Use Claude to design the MCP server tool contracts, the JSON payload schema (what every weekly report must contain), and the HTML template (what slots exist, what data goes where, what the page looks like). High-leverage AI work: the contracts get reused every week. Claude is in the design loop, not the runtime.

01

Weekly trigger n8n on Docker

Scheduled workflow runs every Monday morning. n8n on Docker handles orchestration. Trigger is deterministic — same time, same flow, no LLM in the dispatcher.

02

Direct MCP tool calls n8n → MCP servers

n8n calls each MCP server's tools as structured HTTP endpoints. gsc_query_performance, ga4_compare_period, etc. — no LLM in the data-fetch path. The MCP servers were originally built for Claude to use; once contracts stabilized, n8n can hit them directly. They're just structured APIs at this point.

03

Analysis prose generation Claude API · Slot fill

n8n passes raw weekly data to a Claude API call with a structured-output schema. Claude returns prose for specific slots: "what changed this week and why," "anomaly explanations," "recommended actions." The output schema is fixed — Claude can't invent new sections, only fill the slots the template expects. AI judgment lives here, but inside locked boundaries.

04

JSON composition n8n

n8n combines raw data + Claude prose into a single JSON payload conforming to the design-time schema. Every field is positional, every section is named. If a field is missing, the build fails — no half-rendered reports.

05

GitHub commit n8n GitHub node

JSON pushed as a commit to the GitHub repo. Version-controlled by definition — every weekly report's data lives in git history, auditable, diffable. Pages rebuild triggers automatically.

06

Static template renders GitHub Pages

The HTML template lives in the same repo. On page load it fetches the JSON and populates slots. Layout never moves. Same template every week. Reviewers know exactly where the "what changed" section is, where the action items list is, where the metrics tile sits — every time.

07

Distribution Single URL

Stakeholders bookmark one URL. No email attachment, no shared drive folder, no version conflicts. The report exists where the data is fresh, and looks identical week to week — which is the entire point.

The engineering call
Where AI helps the system, where it hurts it

LLMs are non-deterministic. That's a feature for analysis — different angles each week, fresh framing, language that adapts to what actually changed. It's a bug for layout — stakeholders need to scan the same report shape every Monday morning, not relearn where things are.

The naive AI-First answer is: put Claude everywhere, let it fetch, analyze, render. V1 did this. The reports came out subtly different every week. The bug wasn't analysis quality — it was the rendering loop.

V2 separates the two. AI lives where its non-determinism helps (analysis prose, "what changed and why"). Determinism is locked where stakeholders need it (layout, structure, section order, JSON schema). The MCP servers — originally built FOR Claude — turn out to be reusable as plain APIs once their contracts are stable. n8n calls them directly.

The senior version of "AI-First" isn't AI everywhere. It's knowing which parts of the system AI improves and which parts AI breaks — and architecting the boundary deliberately.

Synthesis

Three different shapes. One pattern underneath.

The cases were picked to stress different shapes of thinking — branching for creative production, linear with feedback for research, hub-and-spoke for analytics. The shapes are different because the problems are different.

What stays constant is the design discipline:

Identify
The actual bottleneck — not "we should use AI more," but the specific point in the existing process where time, accuracy, or throughput breaks.
Design
The contracts between stages. Each stage has a clear input/output. Failures isolate. Stages stay swappable.
Route
The right tool to the right job — including knowing when "no AI at runtime" is the right tool. Flux Pro for hero fidelity. GPT Image for Vietnamese typography. Seedance for motion. Claude API for design-time architecture and runtime slot-fill prose. n8n for deterministic plumbing where layout cannot move. The decision is which to use, and where.
Build
Infrastructure that makes the system queryable. MCP servers, custom skills, structured prompts — the difference between using AI and architecting around it.
Gate
Human judgment where stakes demand it. Domain experts on regulatory content. Taste on creative cuts. AI is upstream of decision, not a replacement for it.

The pattern isn't "use AI." The pattern is: see the system, design the joints, route the right tool to the right job at each joint, and build the infrastructure that makes the whole thing queryable instead of hardcoded.

That is what I would do at PATI in week one — and in week fifty.
